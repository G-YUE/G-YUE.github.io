字符编码

发表于 2017-04-05 | 分类于 python

0000 0000=8位=1字节

字符编码发展史：

ASCII码：127个数字：7个比特位  ----->美国

扩展ASCII：256个数字：8个比特位  ----->拉丁美洲

中文扩展（GB2312）：几千个比特位  ----->中国

中文扩展（GBK）：两万个比特位 ----->中国

Unicode（明文对应二进制数字）：两个字节----六万多个状态---->全世界

utf-8（二进制对应二进制）：可变，字母占1个字节，中文基本上占3个字节。

以文本编辑器为例子，了解下文件存储和执行中的变化。

文件存储：

在文本编辑器中，输入文字、字符或者数字等信息。

在还未点保存之前，数据是存储在内存中，以unicode的形式存在。

当点击保存后，会以文本编辑器规定的编码方式如（utf-8），编码成bytes字节类型的数据存储到硬盘中。

执行中：

文件存储在硬盘中是以某种编码格式形成的bytes字节类型的数据，它会被以原来编码的格式来解码成unicode类型数据放到内存中。

然后被CPU调用执行、输出就会涉及到输出端，使用的编码类型是什么了。
如果不同类型则会乱码。

例如：

我用python编写了一个以UTF-8编码格式的文件，输出“你好”。

保存后，会以UTF-8格式保存到硬盘中。

然后在CMD命令行中执行，它会先从硬盘中以“UTF-8”解码成unicode到内存.

CPU调用，在CMD中显示，但是CMD默认的是GBK编码，所以屏幕中会出现乱码，而不是“你好”。

反之，如果我是用“GBK”编码的python文件，在CMD中就可以正常输出“你好”了。

python 2.X:

1.str:bytes数据

2.unicode:unicode编码后的二进制数据

python 3.x:

1.str:unicode

2.bytes:bytes


python3 把字节和字符类型完全分开了。混用就报错。

数据显示的关键在于编码和解码。

decode :解码 bytes ——–> unicode 以当初编码使用的编码方式，如“UTF-8”，来解码成unicode

encode :编码 unicode ——–> bytes 以任何编码方式，“UTF-8”、“GBK”来编码成字节类型的二进制数

注意点： 

1.unicode、utf-8、GBK都只是编码规则。

2.为什么内存存储unicode.unicode是万国码，可以被编码成多种格式，

3.decode和encode都发生在执行中。执行前都只是数据。

python3:

所有字符在内存中都是unicode
解释器读取的文件的默认编码是utf-8

编码，会把unicode转化成相应的编码的同时，把字符编程bytes

在python2中，str和bytes是一样的。在python3里。str和bytes分开了。str==unicode

数据存到硬盘上，或者网路发送，都必须是bytes格式。
